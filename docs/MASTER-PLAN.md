## 0. TL;DR

**Alfa MP** — мультиплеер-платформа для **GTA 5** (только GTA 5, RDR2 не поддерживаем). Технический фундамент — форк open-source CitizenFX (база FiveM, лицензия BSD-3). Цель — построить продукт уровня RAGE MP, но **превосходящий его** по качеству синхронизации транспорта (главная слабость всех существующих GTA 5 MP-модов) и удобству разработки серверных скриптов.

**Команда:** Owner (стратегия, маркетинг, тестирование, сообщество) + Claude Opus 4.7 (весь код, DevOps, инструментарий, документация). **Без штатных реверс-инженеров** — engine hooks берём готовые из апстрима FiveM.

**Срок до публичного релиза:** 5–7 месяцев активной работы.
**Стартовый бюджет инфраструктуры:** $20–50/мес.
**Производственный бюджет:** $200–500/мес.

**Главные технические преимущества над RAGE MP:**
1. 60 Hz tickrate сервера (vs 30 Hz)
2. State-authoritative vehicle sync (vs owner-authoritative)
3. Soft reconciliation вместо телепортов (snap)
4. Server-side rewind для hit-registration (CS:GO-уровень)
5. Velocity/steering/throttle sync (vs только position+rotation)
6. Driver handoff protocol — без подбросов при пересадках
7. Server-arbitrated collision resolution
8. Server-side анти-чит (валидация движения, спавна, урона)
9. TypeScript-first SDK с hot-reload (vs Lua/C# в RAGE MP)

## 1. Видение

### Что мы строим
Платформу, на которой:
- **Хостеры** (мы + комьюнити) запускают GTA 5 мультиплеер-серверы любого жанра (RP, freeroam, drift, deathmatch)
- **Игроки** подключаются через один наш лаунчер, видят каталог серверов, играют
- **Разработчики ресурсов** пишут серверные скрипты на TypeScript / C# / Lua с современным тулингом

### Для кого

| Сегмент | Что получает |
|---|---|
| **RP-серверы** | Лучший движок для крупных RP-проектов (NoPixel-уровня): стабильная синхра, мощный SDK |
| **Игроки** | Плавная езда, честные хит-реги, единый лаунчер вместо зоопарка из RAGE MP/FiveM/alt:V |
| **Скриптеры** | Современный DX: TypeScript из коробки, hot-reload, нормальный debugger, IDE-поддержка |
| **Хостеры** | Готовое решение поднять сервер за 5 минут, попасть в общий каталог |

### Чем отличаемся (Unique Selling Points)

1. **Заметно плавнее транспорт** — главный визуальный аргумент. Если игрок проедет в нашем форке и в RAGE MP одинаковую трассу — разница будет заметна невооружённым глазом.
2. **Современный стек скриптинга** — TypeScript с проверкой типов вместо Lua-динамики; современный CLI; пакетный менеджер; hot-reload.
3. **Один лаунчер для всех серверов** — UX-уровня Steam, не «скачайте 3 разных клиента под разные моды».
4. **Прозрачная инфраструктура** — открытый roadmap, чёткие метрики аптайма, нет «чёрного ящика» как у RAGE MP.

## 3. Почему форк FiveM (а не с нуля и не другая база)

### Альтернативы и почему отвергнуты

| Вариант | Почему нет |
|---|---|
| **С нуля свой движок** | Нужны 2–3 реверс-инженера на 3–5 лет full-time. Не делаем без RE-команды. |
| **Форк alt:V** | Ядро alt:V **закрытое**, форкнуть нельзя |
| **Форк RAGE MP** | Полностью закрытый исходник, форка не существует |
| **Свой клиент на alt:V SDK** | Возможно, но мы зависим от их инфры и решений |
| **Использовать слитый исходник GTA 5** | IP-кража, гарантированный иск от Take-Two |
| **Форк CitizenFX (FiveM)** ✅ | **Open-source под BSD-3**, рабочая база, есть всё что нужно |

### Что мы получаем «бесплатно» из апстрима FiveM

1. **Engine hooks** в GTA5.exe (entity pool, ped manager, vehicle manager, animation system, weapon system, native registry) — это работа 5+ лет команды реверс-инженеров
2. **Anti-tamper bypass** для Arxan-обфускации Rockstar
3. **Pattern scanner** с автоматическим обновлением сигнатур после Rockstar-патчей
4. **Native invoker** для ~6000 нативов с типами
5. **Asset streaming pipeline** (RPF, YDR, YDD, YTD)
6. **OneSync** — расширенная синхронизация до 1024 игроков
7. **Scripting runtimes** — Lua, JS (V8), C# (.NET)
8. **DirectX 11 хук, IMGUI оверлей**
9. **Build system** (CMake + Premake) уже настроена

### Что мы делаем сами поверх форка

1. **Брендинг** (имя, лого, splash, иконки) — всё своё
2. **Лаунчер** — пишем с нуля (FiveM-лаунчер не open-source в нужной нам форме)
3. **Мастер-сервер** — наш реестр серверов вместо инфры Cfx
4. **Backend** — аккаунты, регистрация, дашборд
5. **Netcode patches** — наши улучшения синхронизации поверх FiveM
6. **Anti-cheat layer** — server-side валидация и эвристики
7. **Modern SDK** — TypeScript-обёртки, тулинг, hot-reload

### 6 основных компонентов

#### 1. Client (форк CitizenFX, ребрендинг Alfa MP)
**Что делает:** инжектится в GTA5.exe игрока, превращая её в мультиплеерную. Подключается к game-серверу по UDP, рендерит других игроков, синхронизирует мир.

**Технологии:** C++ (95%), C# (CitizenFX.Core), Lua/JS/C# runtimes.

**Что меняем по сравнению с upstream:**
- Брендинг (имя, лого, splash, цвета UI)
- Netcode patches (новые поля в snapshot, новый interpolation, reconciliation)
- Crash reporter (Sentry SDK)
- Connect-protocol к нашему мастер-серверу

**Что НЕ трогаем:**
- Engine hooks
- Anti-tamper bypass
- Pattern scanner
- Native registry

#### 2. Launcher (Tauri + React)
**Что делает:** даёт пользователю UI для входа, выбора сервера, обновления клиента и запуска игры.

**Технологии:** Rust (Tauri framework) + TypeScript/React (UI).

**Почему Tauri, а не Electron:**
- Меньший размер (~5 МБ vs ~100 МБ у Electron)
- Меньше памяти (~50 МБ vs ~200 МБ)
- Использует системный WebView2 (уже на Windows 10+)
- Лучше безопасность (Rust-ядро)

**Основные функции:**
- Регистрация / вход (email + пароль, либо Telegram OAuth)
- Server browser (запрос к мастер-серверу)
- Поиск, фильтры, сортировка серверов
- Скачка клиента с CDN при первом запуске
- Авто-обновление клиента
- Запуск GTA5.exe с инжектом и параметром коннекта
- Логи и crash reports

#### 3. Master Server (Node.js + TypeScript)
**Что делает:** ведёт каталог зарегистрированных game-серверов. Все серверы стучат сюда heartbeat'ом, все лаунчеры запрашивают отсюда список.

**Endpoints:**
- `POST /v1/servers/heartbeat` — game-сервер шлёт пинг каждые 30 сек
- `GET /v1/servers/list` — лаунчер берёт список активных серверов
- `POST /v1/servers/register` — регистрация нового сервера (выдача API-ключа)
- `GET /v1/servers/{id}` — детальная инфа сервера

**Технологии:** Node.js + TypeScript + Fastify + PostgreSQL + Redis (для активных серверов в кэше).

#### 4. Backend (Node.js + PostgreSQL)
**Что делает:** аккаунты пользователей, регистрация хостеров, админ-дашборд.

**Модули:**
- User registration / login (Argon2 пароли, JWT-токены)
- Server owner registration (после ручной модерации в начале)
- Admin dashboard (React, отдельный домен `admin.alfamp.com`)
- Banlist (UUID игроков, IP, hardware ID)
- Audit log (кто что сделал)
- Statistics (DAU/MAU, серверы, ошибки)

#### 5. Game Server (форк FXServer + наши улучшения)
**Что делает:** конкретный игровой сервер, на котором играют игроки. Хостится либо нами (officials), либо комьюнити.

**Технологии:** C++ (ядро FXServer), V8/Mono/Lua (скрипты ресурсов).

**Что улучшаем:**
- 60 Hz tickrate по умолчанию (vs 30)
- Расширенный snapshot формат
- Server-side rewind для hit-confirm
- Server-side anti-cheat валидация
- TypeScript-first SDK для разработки ресурсов

#### 6. CDN (Cloudflare R2 или Bunny.net)
**Что делает:** раздаёт клиент-билды, ассеты, обновления.

**Технологии:** Cloudflare R2 (S3-compatible, без egress fees) или Bunny.net (быстрее в РФ/СНГ).

## 5. Главные технические улучшения vs RAGE MP — Vehicle Sync Deep Dive

Это **главный маркетинговый аргумент** Alfa MP. Детально распишу что и как.

### Проблема — почему машины в RAGE MP/FiveM «прыгают»

1. **Физика RAGE не детерминирована** — два ПК с идентичными входами симулируют немного разную траекторию из-за floating-point precision, разных тиков, разных CPU
2. **Owner-authoritative модель** — только водитель видит «настоящую» физику, остальные интерполируют с задержкой
3. **Снепшот содержит только pos+rot** — невозможно нормально экстраполировать без velocity
4. **Резкий snap при коррекции** — клиент видит «правильную позицию» и телепортирует машину туда без сглаживания
5. **При коллизии у одной из машин нет authority** — она «реагирует» на удар уже после факта
6. **При пересадке authority мгновенно мигрирует** — пакеты теряются, машина дёргается

### Решение № 1: Расширенный snapshot

**Текущий FiveM snapshot для машины:**
```typescript
{
  netId: u32,
  position: Vec3,        // x, y, z
  rotation: Quat,        // pitch, yaw, roll
  health: u16
}
```

**Наш расширенный snapshot:**
```typescript
{
  netId: u32,
  position: Vec3,
  rotation: Quat,
  velocity: Vec3,        // НОВОЕ: для экстраполяции
  angularVelocity: Vec3, // НОВОЕ: для вращения колёс
  steering: f32,         // НОВОЕ: угол руля (-1..1)
  throttle: f32,         // НОВОЕ: газ (0..1)
  brake: f32,            // НОВОЕ: тормоз (0..1)
  handbrake: bool,       // НОВОЕ: ручник
  gear: u8,              // НОВОЕ: передача
  rpm: f32,              // НОВОЕ: обороты
  damage: VehicleDamage, // НОВОЕ: детальные повреждения
  occupants: u32[],      // ID седоков
  inputTimestamp: u64,   // НОВОЕ: время на стороне owner'а
  health: u16
}
```

**Зачем каждое поле:**
- `velocity` — позволяет экстраполировать позицию ahead 100–200 мс без видимого jitter
- `steering` — другие клиенты могут предсказать траекторию поворота
- `throttle/brake` — предсказание ускорения/торможения
- `inputTimestamp` — реконсилиация на правильный момент времени

**Стоимость:** +40 байт на snapshot (с 30 до 70 байт). При 60 Hz и 64 игроках = +5 КБ/с трафика, незначительно.

### Решение № 2: Soft reconciliation

**Сейчас (FiveM):** клиент видит «правильную позицию», машина snap'ается туда мгновенно → видимый прыжок.

**У нас:**
```
1. Клиент получает корректирующий snapshot от сервера
2. Сохраняет текущую (предсказанную) позицию как `from`
3. Сохраняет серверную как `to`
4. Над следующие 200 мс плавно интерполирует `from` → `to`
5. Если в этот момент пришёл ещё один snapshot — обновляем `to` без рестарта анимации
```

Эффект: **−90% видимых телепортов** при той же серверной нагрузке.

### Решение № 3: Velocity-aware extrapolation

**Сейчас (FiveM):** между snapshot'ами клиент линейно экстраполирует по последним 2 позициям → плохо работает в поворотах.

**У нас:**
- Используем `velocity` + `angularVelocity` для физически корректной экстраполяции
- Если `throttle > 0` — добавляем предполагаемое ускорение
- Если `brake > 0` — добавляем торможение
- Колёса вращаются согласно `velocity.length() / wheelRadius`
- Корпус качается согласно `angularVelocity`

**Эффект:** машина едет **визуально плавно** между snapshot'ами, особенно на скорости 80+ км/ч.

### Решение № 4: State-authoritative модель

**Сейчас (FiveM, OneSync):** owner-authoritative с миграцией. Каждая машина имеет «владельца» (обычно ближайший игрок), он шлёт state серверу, сервер ретранслирует.

**Проблемы owner-authoritative:**
- При миграции authority — момент рассинхрона
- Owner может читерить (модифицированный клиент шлёт ложный state)
- Server не знает «настоящего» состояния — только верит клиенту

**Наша модель — state-authoritative:**
1. Клиент-owner шлёт **inputs** (steering, throttle, brake), не state
2. Сервер запускает упрощённую физическую модель машины (без полного RAGE-движка — это невозможно, но кинематическую)
3. Сервер вычисляет «правильную» позицию и шлёт всем
4. Все клиенты (включая owner) reconcile к серверу

**Стоимость:** +20% CPU сервера на vehicle simulation. Решаемо, на современном CPU 64 игроков нормально.

**Эффект:**
- Невозможно классическое spoof-читерство (телепорт машины)
- Идентичные траектории у всех клиентов
- Решение проблемы миграции authority

### Решение № 5: Driver handoff protocol

**Сейчас (RAGE MP/FiveM):** игрок жмёт F → клиент сразу садится → пакет идёт серверу → сервер обновляет owner'а → разница состояний → подброс машины.

**У нас:**
```
1. Игрок жмёт F → клиент шлёт CLIENT_REQUEST_ENTER_VEHICLE
2. Сервер ставит машину в state PENDING_ENTRY (физика паузится на 50 мс)
3. Сервер бродкастит всем VEHICLE_PENDING_ENTRY {entityId, playerId}
4. Все клиенты видят анимацию «садится», машина зафиксирована
5. Через 50 мс сервер запускает физику с новым owner'ом
6. Никаких подбросов
```

**Эффект:** **полностью убирается** «прыжок машины при посадке» — главный визуальный баг всех GTA 5 MP-модов.

### Решение № 6: Collision arbitration

**Сейчас:** при столкновении двух машин у одной authority, у другой нет. У authority физика играет, у второй — догоняет. В момент удара — desync, машина может улететь в небо.

**У нас:**
1. Обе машины шлют серверу `COLLISION_DETECTED {with, point, impulse}` в течение 10 мс
2. Сервер сравнивает оба отчёта, выбирает «ground truth» (на основе позиций до удара)
3. Сервер вычисляет реакцию обеих машин на основе их масс, скоростей
4. Шлёт обеим клиентам `COLLISION_RESOLVED {vehicleA_state, vehicleB_state}`
5. Оба клиента soft-reconcile к серверу

**Эффект:** **−60% улётов в стратосферу** при столкновениях.

### Решение № 7: Server-side rewind для hit-registration

Это **отдельная от vehicle sync** фича, но в этой же категории — улучшение синхры.

**Сейчас (RAGE MP/FiveM):** клиент стреляет → шлёт серверу «попал в игрока X в точке Y» → сервер доверяет (или нет). При пинге 80 мс игрок Y уже сдвинулся на 1.5 м → попадание не засчитывается.

**У нас — Valve/CS:GO-стиль:**
1. Клиент стреляет в момент `T_local`
2. Шлёт серверу `SHOT {shooterPos, direction, T_local}`
3. Сервер берёт snapshot мира на момент `T_local` (из ring buffer 1 сек)
4. Делает ray-cast в этом «прошлом» мире
5. Если попал — наносит урон
6. Шлёт результат

**Эффект:** **>95% hit-accuracy** при пинге до 150 мс. Игроки чувствуют, что попадания «честные» — это огромный фактор удержания в шутерах и RP-режимах.

### Решение № 8: Adaptive tickrate per entity

**Сейчас (FiveM):** все entity синкаются одинаково — 30 Hz.

**У нас:**
- **Машины в радиусе 50 м от тебя:** 60 Hz, полный snapshot
- **Машины 50–200 м:** 30 Hz, базовый snapshot
- **Машины 200–500 м:** 10 Hz, только позиция
- **Машины >500 м:** не синкаются (scope-out)

**Эффект:** ~70% экономия трафика без видимой потери качества. Позволяет 60 Hz основного цикла без перегрузки сети.

## 8. Инфраструктура и DevOps

### Среды (environments)

| Среда | Назначение | Где живёт |
|---|---|---|
| `local` | Разработка | Локальный ПК (после установки VS) |
| `dev` | Интеграционная сборка | Hetzner CX22, $5/мес |
| `staging` | Pre-release бета | Hetzner CX32, $10/мес |
| `production` | Прод | Hetzner Dedicated, $50–200/мес после релиза |

### CI/CD

**GitHub Actions workflows:**

1. `client-build.yml` — собирает Windows-клиент при пуше в `main`. Артефакты → CDN.
2. `master-server-deploy.yml` — деплоит мастер-сервер на VPS при пуше.
3. `game-server-deploy.yml` — деплоит обновлённый game-server на наш officials.
4. `launcher-build.yml` — собирает лаунчер (Tauri) под Windows.
5. `release.yml` — на тэг `v*` создаёт GitHub Release со всеми артефактами.

**Стратегия версионирования:**
- SemVer: `1.0.0`, `1.0.1`, `1.1.0`, `2.0.0`
- Каждый коммит в main = bump patch (`1.0.0` → `1.0.1`)
- Merge feature branch = bump minor
- Breaking change = bump major (редко)

### Backup strategy

**PostgreSQL:**
- Hourly `pg_dump` на VPS → S3 (Backblaze B2, $0.005/ГБ)
- Daily snapshot всего VPS через Hetzner Snapshot ($1/мес)
- Retention: 7 daily, 4 weekly, 3 monthly

**Game-server world state:**
- Сохраняется в БД сервера каждые 5 минут
- Backup как у master-сервера

**Disaster recovery:** RTO 1 час, RPO 1 час.

### Мониторинг

**Что мониторим:**
- Аптайм мастер-сервера (Better Stack, проверка каждые 30 сек)
- Аптайм каждого officials game-сервера
- CPU/RAM/disk на VPS (Prometheus node exporter)
- Server tickrate (custom metric из FXServer)
- Player count online (real-time)
- Error rate в Sentry
- API latency мастер-сервера (p50/p95/p99)

**Алерты в Telegram:**
- Сервер down >2 мин
- Tickrate упал ниже 45 Hz на >5 мин
- Error rate >5% за 5 мин
- Диск заполнен >85%

### Security

**Что защищаем:**
- Аккаунты пользователей: Argon2id хеши, rate-limit на login (5 попыток / 15 мин)
- API ключи серверов: ротация по требованию, hash в БД
- TLS везде: Let's Encrypt, auto-renew
- DDoS-защита: Cloudflare перед мастер-сервером и backend
- Game-сервер UDP: anti-flood на L4 (rate-limit per IP)

**Что НЕ делаем (сознательно):**
- Не пишем свой kernel-level anti-cheat (BattlEye-уровень) — слишком дорого
- Не делаем 2FA в первом релизе (добавим позже)

## 13. Риски и митигации

### Высокие риски

| Риск | Вероятность | Impact | Митигация |
|---|---|---|---|
| **Take-Two подаст иск** | Low (20%) | Critical | Кристальный ребрендинг, без ассетов Rockstar, юрист на консультации до запуска |
| **Rockstar выкатит крупный патч → проект встанет на 1–2 недели** | High (90% в течение года) | Medium | Стягиваем FiveM-апстрим автоматически, готовы держать паузу |
| **Маркетинг провалится → никто не придёт** | High (50%) | Critical | План B: партнёрство с существующим RP-проектом, который перейдёт на наш стек |
| **Vehicle sync не выйдет лучше FiveM** | Medium (30%) | High | Тестируем на промежуточных метриках с M3, переоцениваем подход если не выходит |
| **Owner выгорит на месяце 4–5** | Medium (40%) | Critical | Чёткие маленькие победы каждую неделю, разделение труда, не давать овертайма |

### Средние риски

| Риск | Митигация |
|---|---|
| FiveM-апстрим сделает breaking change | Pinned версия, ручной merge с оценкой |
| Discord-комьюнити токсичное | Жёсткая модерация с дня 1 |
| Дешёвый VPS не выдержит нагрузку | План масштабирования: Hetzner Dedicated в один клик |
| Деньги закончатся раньше выручки | Заранее план: или донаты от ранних фанов, или личный bridge-funding |

### Низкие риски

| Риск | Митигация |
|---|---|
| GitHub забанит репо | Параллельный backup на GitLab |
| Cloudflare заблокирует CDN | Backup на Bunny.net |
| Claude недоступен (Anthropic down) | Резервная разработка вручную (медленно) |

## 16. Глоссарий

| Термин | Что значит простыми словами |
|---|---|
| **CitizenFX** | Open-source основа FiveM, которую мы форкаем |
| **FXServer** | Игровой сервер из FiveM-экосистемы |
| **Tickrate** | Сколько раз в секунду сервер пересчитывает состояние мира (30Hz / 60Hz) |
| **Snapshot** | Срез состояния мира, который сервер шлёт клиентам |
| **Reconciliation** | Сглаживание разницы между предсказанием клиента и правдой сервера |
| **Authority** | Кто решает «настоящее» состояние — клиент или сервер |
| **OneSync** | Расширенный режим FiveM для серверов >32 игроков |
| **Native** | Функция, экспортированная движком RAGE (всего ~6000 штук) |
| **Hook** | Перехват вызова функции в чужом бинаре для модификации поведения |
| **RPF** | Формат ассет-архива RAGE (наш CDN раздаёт распакованные/упакованные RPF) |
| **Heartbeat** | Периодический пинг от game-сервера мастер-серверу |
| **Master server** | Центральный реестр всех игровых серверов |
| **CDN** | Сеть доставки контента — раздаёт клиент-билды быстро и дёшево |
| **VPS** | Виртуальный сервер в облаке (Hetzner, OVH) |
| **CI/CD** | Автоматизация сборки и деплоя при изменении кода |
| **Sentry** | Сервис ловли крашей и ошибок |
| **Grafana** | Графики метрик в реальном времени |
| **MSVC** | Компилятор Microsoft (Visual Studio) для C++ |
| **Tauri** | Фреймворк для маленьких десктопных приложений на Rust + Web |
